## Δ) 2FA — Επιβεβαίωση δύο βημάτων (κωδικός μιας χρήσης στο κινητό του ιδιοκτήτη)
Χτίστηκε και στα 3 προγράμματα (07/07/2026). Ρέει: login → οθόνη κωδικού → κωδικός → είσοδος.
Ισχύει για όλους πλην admin/guest. Μόνο σε φρέσκο login (μετά από πλήρη έξοδο), όχι σε refresh
(κρατιέται σημάδι στο sessionStorage). Αρχεία: `twoFactor.js` (client) + `netlify/functions/two-factor.js` (server).

**Κατάσταση:** στο DEV ο κωδικός εμφανίζεται στην οθόνη (κίτρινο πλαίσιο). Στο PROD στέλνεται SMS.

**⚠️ ΠΕΡΙΟΡΙΣΜΟΣ ΑΣΦΑΛΕΙΑΣ (να αποφασιστεί):** το 2FA όπως είναι προστατεύει την ΟΘΟΝΗ, όχι τη
βάση. Όποιος έχει τον κωδικό (password) μπορεί τεχνικά να μιλήσει απευθείας στη βάση παρακάμπτοντας
την οθόνη — γιατί το password μόνο του δίνει έγκυρο token. Καλύπτει τον «κλεμμένο αποθηκευμένο
κωδικό από μη-τεχνικό» ΟΧΙ τον έμπειρο hacker. Για πλήρη κάλυψη χρειάζεται μεγαλύτερη αλλαγή:
custom-token auth (function ελέγχει password+κωδικό ΚΑΙ μετά δίνει το «κλειδί»· απενεργοποίηση
του email/password direct sign-in). ΕΚΚΡΕΜΕΙ απόφαση χρήστη αν θα προχωρήσουμε σε αυτό.

**ΣΧΕΔΙΟ «ΑΔΙΑΠΕΡΑΣΤΟ» 2FA (εγκρίθηκε από χρήστη 07/07 — να γίνει ΠΡΟΣΕΚΤΙΚΑ στο τεστ, ΟΧΙ βιαστικά, ΟΧΙ μαζί με go-live):**
Στόχος: να μη φτάνει το password μόνο του για πρόσβαση στα δεδομένα (κάλυψη και έναντι έμπειρου).
Αρχιτεκτονική (custom-token auth):
1. Netlify function `login`: παίρνει {username, password, code}. Επαληθεύει password (server-side
   `signInWithPassword`) + εξαψήφιο. Αν ΟΚ, φτιάχνει **Firebase custom token** (με το service account)
   με claims: identity (email σε ΜΗ-δεσμευμένο όνομα, π.χ. `vem`), role, seller, `twofa:true`.
2. Client: `signInWithCustomToken(token)` (SDK στο app· REST `accounts:signInWithCustomToken` σε eidikes/installations).
3. **Rules rewrite:** όλα τα `auth.token.email` → `auth.token.vem`. Έτσι μια απευθείας είσοδος με σκέτο
   password (χωρίς `vem`) απορρίπτεται από τη βάση. Admin: περνάει κι αυτός από τη function (χωρίς
   βήμα κωδικού) ώστε να πάρει `vem`.
ΠΡΟΣΟΧΗ/ρίσκα: (α) δεν δοκιμάζεται με `npm run web` — θέλει `netlify dev` + service account JSON τοπικά.
(β) λάθος στα rules → κλείδωμα όλων ή άνοιγμα όλων. (γ) ΑΒΕΒΑΙΟ αν το custom-token ID token φέρει
`auth.token.email` — γι' αυτό ΠΡΩΤΟ ΒΗΜΑ: εμπειρική δοκιμή (mint custom token → decode) πριν αγγίξουμε rules.
(δ) εφαρμογή rules σε test ΚΑΙ στις 2 βάσεις παραγωγής (vaiconcloud + vaicon-eidikes) στο go-live.

**GO-LIVE 2FA (να ΜΗΝ ξεχαστεί):**
1. Netlify env vars σε ΚΑΘΕ site: `OWNER_PHONE` (κινητό ιδιοκτήτη), `YUBOTO_API_KEY`,
   `YUBOTO_SENDER`, `YUBOTO_TEST_MODE=false`. (τα `FIREBASE_SERVICE_ACCOUNT`/`FIREBASE_DB_URL`
   υπάρχουν ήδη σε app/eidikes· ΟΧΙ ακόμα στο installations — βλ. σημείο 2.)
2. **vaicon-installations:** δεν είχε καθόλου Netlify functions. Προστέθηκαν `two-factor.js` +
   `lib/fbAdmin.js`, αλλά ΘΕΛΕΙ στήσιμο env στο Netlify site (service account + DB url + Yuboto)
   πριν δουλέψει live. **ΕΚΚΡΕΜΕΙ.**
3. Firebase rules: ο κόμβος `/twofa` ΔΕΝ πρέπει να είναι αναγνώσιμος από χρήστες (καμία `.read`
   → μπλοκάρεται· ο server μπαίνει μέσω service account). Έλεγχος πριν το live.
4. Μία δοκιμή με αληθινό SMS μετά το πρώτο deploy.
