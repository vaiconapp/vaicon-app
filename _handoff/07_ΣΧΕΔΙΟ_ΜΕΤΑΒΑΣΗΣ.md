## Στόχος / απόφαση
- Και τα 3 προγράμματα να δουλεύουν ζωντανά σε ΜΙΑ κοινή βάση: **`vaicon-eidikes`** (όπως ήδη συνεργάζονται στο `vaicon-test`).

- **Ειδικές + Τοποθετήσεις: καμία αλλαγή βάσης.** Το `vaiconcloud` εγκαταλείπεται.

## Δεδομένα στη μετάβαση (επιβεβαιωμένο από τον χρήστη)
- Ό,τι υπάρχει στο `vaicon-test` και στα 3 προγράμματα είναι **δοκιμαστικό = σκουπίδια**. ΔΕΝ μεταφέρεται πουθενά· η `vaicon-test` εγκαταλείπεται.
- **Ειδικές + Τοποθετήσεις:** κρατούν τα ζωντανά τους δεδομένα (ήδη στο `vaicon-eidikes`). Καμία αλλαγή.
- **Τυποποιημένες:** ξεκινούν **εντελώς άδειες** (μηδέν παραγγελίες) — απλώς δείχνουν στην κοινή βάση.
- **Κοινές βάσεις** (πελάτες/κλειδαριές/επενδύσεις): υπάρχουν ήδη ζωντανές στο `vaicon-eidikes`.
- ΠΡΟΣΟΧΗ — ρυθμίσεις που ζουν ΜΕΣΑ στη βάση και ήταν δοκιμαστικές, θα **ξαναγίνουν μία φορά πάνω στα ζωντανά**:
  1. Ανάθεση **πωλητή ανά πελάτη** (πεδίο `seller` στους πελάτες).
  2. **Δικαιώματα έγκρισης** (`approval_rights`, από μενού admin).

## ΒΗΜΑ 2α — Κουπόνι αρίθμησης (μηδέν διπλά νούμερα σε ταυτόχρονη έγκριση)
ΓΙΑΤΙ ΕΔΩ & ΟΧΙ ΣΤΟ TEST: στο κοινό `vaicon-test` οι κανόνες είναι ορθάνοιχτοι (`.write: auth!=null` στη ρίζα). Στη Firebase ένας «true» κανόνας γονέα ΔΕΝ ακυρώνεται από παιδί, οπότε το «μόνο-δημιουργία» δεν μπορεί να επιβληθεί εκεί χωρίς να ξαναγραφτεί όλη η δομή (ρίσκο για τις άλλες 2 εφαρμογές που μοιράζονται τη βάση). Άρα μπαίνει στους ανά-εφαρμογή κανόνες της ζωντανής `vaicon-eidikes`, μαζί με το ΒΗΜΑ 2.

**Κανόνας (προσθήκη στο `order_seq` της ζωντανής `vaicon-eidikes`):**
```json
"order_seq": {
  ".read": "<ίδιο με special_orders: authed μη-guest>",
  ".write": "auth.token.email === 'admin@vaicon.local'",
  "$num": {
    ".write": "auth != null && !data.exists() && newData.val() != null"
  }
}
```
- Ο γονέας `.write` = ΜΟΝΟ admin → επιτρέπει το restore (γράφει όλο το node). Δεν είναι «true», άρα ΔΕΝ δίνει δικαίωμα overwrite στους απλούς χρήστες.
- Το `$num` = «γράφε μόνο αν ΔΕΝ υπάρχει» → κουπόνι: όποιος προλάβει κλειδώνει το νούμερο, ο δεύτερος αποτυγχάνει και δοκιμάζει το επόμενο.

**Κώδικας (ApprovalScreen.js — και στα 2 προγράμματα):** αντικατάσταση του «βρες επόμενο + PATCH» με claim-loop:
```js
const claimBaseNumber = async () => {
  let n = Number(await nextBaseNumber());
  for (let i = 0; i < 100; i++, n++) {
    const res = await fetch(`${FIREBASE_URL}/order_seq/${n}.json`, { method: 'PUT', body: JSON.stringify(Date.now()) });
    if (res.ok) return String(n);   // κλειδώθηκε
  }                                  // αλλιώς (permission denied) → πιασμένο, επόμενο
  throw new Error('order number claim failed');
};
```
- `approve`: `const number = await claimBaseNumber();` (αφαιρείται το ξεχωριστό PATCH `{[number]:1}`).
- `approveGroup`: `const base = await claimBaseNumber();` (το claim κρατά τη βάση· τα `-1/-2` παράγονται από αυτήν).
- Προσοχή στο auth: στο live το REST fetch χρειάζεται `?auth=<idToken>` (όπως ήδη γίνεται στα άλλα fetch των Ειδικών). Να μπει το ίδιο query string που χρησιμοποιεί το app στη ζωντανή βάση.
- Γραφείο (`CustomScreen.js`, αυτόματη πρόταση νούμερου): να χρησιμοποιήσει το ίδιο `claimBaseNumber` αντί για PATCH. Χειροκίνητο νούμερο από υπάλληλο: μένει ως έχει (ευθύνη γραφείου, υπάρχει ήδη έλεγχος διπλότυπου).

**Δοκιμή (μετά την εφαρμογή κανόνων στη ζωντανή):** δύο ταυτόχρονες εγκρίσεις → δύο διαφορετικά νούμερα. Restore ως admin → περνά κανονικά.
